# MailTrack — Living Project Plan

Status: **Live** (backend deployed 2026-07-08, DB migrated, extension config pointed at production) | Owner: Heytt Satra | Repo: https://github.com/heytt-satra/mailtracker- | API: https://mailtrack-api.heyttsatra17.workers.dev

This document is the single source of truth for MailTrack. It is updated after every meaningful change. Nothing is implemented before it is planned here first.

---

## 1. Executive Summary

MailTrack is a Gmail email-tracking Chrome extension and backend service that reports opens and clicks with **verified accuracy**. Every competitor in this space (Mailsuite, Mailtrack.io, Streak, Yesware) reports a "read" the instant a tracking pixel is fetched — but pixel fetches are triggered by Apple Mail Privacy Protection, notification-preview rendering, and corporate security scanners just as often as by an actual human opening the email. MailTrack's differentiator is a verification engine that classifies every pixel fetch before it is allowed to surface as a "read," and explicitly reports "not verifiable" rather than fabricating engagement when the signal is ambiguous.

## 2. Product Vision

Be the email tracker professionals trust because it tells the truth. Where competitors optimize for impressive-looking open rates, MailTrack optimizes for **decision-grade accuracy** — if it says "opened," the sender can act on that with confidence (follow up, escalate, close). No watermark on the free tier. No fabricated engagement, ever.

## 3. Product Requirements

### 3.1 Functional Requirements

- FR1: User can toggle tracking on/off per email from Gmail compose.
- FR2: Sent emails get an invisible pixel and rewritten trackable links, with no visible watermark or branding.
- FR3: Sent-folder thread view shows a per-email status indicator (Sent / Delivered / Opened / Clicked / Not verifiable).
- FR4: User receives a desktop notification only on verified-open or click events — never on raw pixel fetch.
- FR5: User can view a per-email event timeline showing all fetch events, with suppressed/non-human events visibly greyed out and labeled with the reason they were suppressed.
- FR6: Status only escalates: `Sent → Delivered → Opened (verified) → Clicked`. The system never downgrades or fabricates a status.
- FR7: When opens are structurally unverifiable for a recipient (e.g., detected Apple Mail Privacy Protection relay), the UI explicitly states this instead of showing a false positive or a silent blank.
- FR8: Click tracking redirects through MailTrack and preserves the original destination exactly.
- FR9: User can export event history as CSV.
- FR10: User can delete tracking data for a given sent message (privacy control).

### 3.2 Non-Functional Requirements

- NFR1 (Performance): Pixel endpoint p95 response time < 100ms globally (Cloudflare Workers edge).
- NFR2 (Reliability): No message send is ever blocked or delayed by tracking failure — tracking injection fails open (email sends untracked rather than not sending).
- NFR3 (Security): No tracking token is guessable; all tokens are cryptographically random (128-bit+). No PII beyond what Gmail already exposes is collected.
- NFR4 (Privacy): IP addresses are hashed at rest after ASN/geo classification is derived; raw IPs are not retained beyond 30 days.
- NFR5 (Maintainability): Backend and extension are independently deployable; classifier rules are unit-testable in isolation from the HTTP layer.
- NFR6 (Honesty): The false-positive rate on "Opened, verified" must be validated against real device tests (Phase 3) before any read notification ships to users.
- NFR7 (Scalability): Backend must handle burst traffic (scanner storms hitting the pixel endpoint within milliseconds of send) without degrading legitimate event processing.

### 3.3 User Stories

- As a founder emailing a prospect, I want to know the moment they actually read my email, not the moment their phone glances at it, so I can time my follow-up call correctly.
- As a salesperson, I want to trust my open rate numbers when reporting to my manager, so I don't get caught out by inflated metrics from prefetching.
- As a privacy-conscious user, I want to know when a recipient's platform makes tracking fundamentally impossible, instead of being shown a fake number.
- As a free-tier user, I don't want my emails to look like they came from a tracking tool.

## 5. Technical Architecture

```
Gmail (web) ── Chrome Extension (MV3, WXT, InboxSDK) ── HTTPS ──▶ Cloudflare Worker (Hono)
                                                                        │
                                                        ┌───────────────┼────────────────┐
                                                        ▼               ▼                ▼
                                                 POST /messages   GET /p/:t.gif     GET /l/:t
                                                        │               │                │
                                                        └───────────────┴────────────────┘
                                                                        ▼
                                                          Supabase Postgres (raw events)
                                                                        ▼
                                                       Classifier (async, Worker cron / queue)
                                                                        ▼
                                                              verdicts table ──▶ /events (SSE/poll)
                                                                        ▼
                                                        Extension background worker ──▶ chrome.notifications
```

### Architecture Decisions (ADRs)

- **ADR-1: Cloudflare Workers over a VPS.** Chosen for global edge latency on the pixel endpoint (NFR1) and zero ops burden. Tradeoff: Workers' CPU-time limits mean the classifier runs as a separate async step (Cloudflare Queue consumer or cron trigger), not inline in the pixel handler. This is also *correct* behavior — the pixel response must never be slowed down by classification logic.
- **ADR-2: Supabase Postgres over a KV store.** Event data is relational (messages → events → verdicts) and needs ad-hoc querying for classifier tuning. Supabase gives Postgres + realtime + auth for free-tier scale.
- **ADR-3: InboxSDK over raw DOM scraping.** Gmail's DOM is unstable and obfuscated; InboxSDK is the de facto standard used by Streak, Mixmax, etc., and handles Gmail's SPA re-renders correctly.
- **ADR-4: No-cache pixel response.** `Cache-Control: no-store` on the pixel is required so repeat opens are observable — this is what makes the "verified" escalation ladder possible at all.
- **ADR-5: Verdicts only escalate, never downgrade or get deleted.** This is a product-integrity decision, not just an engineering one: it is the mechanism that makes "we don't fabricate engagement" true rather than aspirational.
- **ADR-6: ASN resolved from `request.cf.asn`, not a MaxMind mmdb lookup.** Cloudflare already resolves the requesting IP's ASN at the edge and exposes it on `request.cf.asn`/`asOrganization` for free — no GeoIP database needed in the hot path. This changes MaxMind's role from "resolve IP→ASN" (original plan) to "help build the `asn_intel` category mapping" (asn→apple_mpp/security_scanner/...), which is a strictly smaller and simpler job. Discovered during Phase 1 implementation; adopted because it removes an entire dependency (mmdb parsing in a Worker) for equivalent accuracy.
- **ADR-7: Polling over SSE for `/v1/events/*`.** True server-push on Cloudflare Workers needs a Durable Object to hold a connection open; without one, an SSE endpoint would be a half-working long-poll wearing an SSE label. Rather than ship that, v1 exposes `GET /v1/events/poll?since=<iso>` and the extension's background worker polls it every few seconds. Real push (Durable Objects, or Supabase Realtime consumed directly by the extension) is tracked in Future Improvements, not silently deferred.
- **ADR-8: Apple Mail Privacy Protection is detected by IP-range containment, not ASN.** Apple Private Relay's second hop can egress through third-party CDN partner ASNs, not just Apple's own — so an ASN→apple_mpp mapping risks misclassifying unrelated traffic that happens to share an ASN with a CDN partner (a false "not verifiable" for a real human, or worse, silently swallowed opens). Apple publishes an authoritative CSV of egress ranges at `mask-api.icloud.com/egress-ip-ranges.csv` (verified reachable during Phase 1 — response exceeds 10MB, consistent with a real, large, current range list). MailTrack stores these in a Postgres `ip_ranges` table and resolves membership via the native `inet <<= cidr` containment operator through a `classify_ip_category()` SQL function, checked before the ASN-based path in the classifier. `asn_intel` remains correct for security scanners, which do run dedicated ASNs.
- **ADR-9: Compose interception uses cancel-then-resend, not "modify in place."** InboxSDK's `presending` event is documented as firing "when the user presses send" with a `cancel()` escape hatch, but is NOT documented to await an async handler before the send proceeds — confirmed by direct research against the InboxSDK docs during Phase 2 (no documented promise/resume support). Modifying the compose body asynchronously inside an unawaited `presending` handler risks a race where Gmail's actual send fires before `setBodyHTML()` has run, shipping the email without the pixel. MailTrack instead calls `event.cancel()` synchronously on first `presending`, performs the tracking injection (or fails open) asynchronously, and then calls `composeView.send()` itself in a `finally` block — with a guard flag so a possible re-fired `presending` from that programmatic resend is a no-op pass-through rather than a second cancel (avoiding an infinite loop, undocumented either way). Also hand-declared the used InboxSDK surface in `inboxsdk-types.ts` rather than trusting `@inboxsdk/core`'s shipped types, which are inconsistent across published versions.
- **ADR-10: Supabase Auth is the signup/login identity gate; MailTrack's own API key remains the only request-auth mechanism.** The original design required manually inserting a `users` row and computing an API key hash by hand — functional for a solo dev, unusable as a real product. Rather than reinventing password auth (hashing, rate limiting, email verification, reset flows — all real security surface Supabase Auth already handles correctly), the extension's options page calls Supabase Auth directly, client-side, with the public anon key (`signUp`/`signInWithPassword`, standard stable API — verified method signatures against Supabase's docs before implementing). The resulting session's access token is then exchanged, exactly once, for a MailTrack API key via a new `POST /v1/auth/provision` endpoint, which validates the token server-side via `supabase.auth.getUser(jwt)` (confirmed real API: takes an access token, returns the user it belongs to or an error) and upserts a `users` row keyed by the Supabase auth user's own id (`users.id references auth.users(id)`, the standard Supabase "profiles" pattern — no redundant second UUID). Every other endpoint's existing, tested, hardened API-key auth middleware is completely untouched — Supabase Auth never becomes the ongoing request-auth mechanism, only the one-time account-creation/key-issuance gate. Each provision call rotates the key (simplest self-serve "I lost my key" recovery for v1, at the cost of invalidating any prior key for the account — an acceptable trade for a credential meant to live in exactly one browser's extension storage).
- **ADR-11: the backend hosts its own email-confirmation landing page rather than relying on Supabase's default.** Discovered live, post-deployment, when the user's own confirmation email redirected to `localhost` and failed — a class of bug that only a real human clicking a real email link can surface, invisible to every unit/integration/smoke test run so far. MailTrack has no ordinary website for Supabase's Auth to redirect to by default (its "Site URL" fallback), so `GET /auth/confirmed` was added to the already-deployed backend (near-zero marginal cost — same Worker, one more route, no new infrastructure) and wired up via `signUp()`'s `emailRedirectTo` option (confirmed real Supabase API before use). The one piece this can't fully automate: Supabase enforces a Redirect URLs allow-list for security, so the exact `https://mailtrack-api.heyttsatra17.workers.dev/auth/confirmed` URL still needs to be added in the Supabase dashboard by hand — no Management API token was available to do this programmatically, and fabricating a workaround (e.g., disabling the allow-list) was rejected as loosening a real security control to save one dashboard click.
- **ADR-12: InboxSDK's MV3 bootstrapping needs three pieces most integration guides gloss over — the `scripting` permission was necessary but not sufficient.** The user reported no tracking data reaching the backend; querying the live database directly (not guessing) confirmed signup/login/API-key issuance all worked, but `messages` was completely empty — the compose hook's `POST /v1/messages` never fired. Root cause, fully traced: InboxSDK's content script runs in an isolated JS world and can't reach Gmail's own page state, so it (1) asks the background service worker via `chrome.runtime.sendMessage({type: 'inboxsdk__injectPageWorld'})` to (2) run `chrome.scripting.executeScript({world: 'MAIN', files: ['pageWorld.js']})`, which (3) requires a file literally named `pageWorld.js` to exist at the extension's build root. Adding the `scripting` permission alone (first attempted fix) only satisfied precondition (2) — the user still hit the identical error afterward, which is what triggered digging further rather than assuming a first plausible-looking fix was the whole story. Traced the rest by reading the actual installed `@inboxsdk/core` package directly (`node_modules/@inboxsdk/core/background.js` — 27 lines, is literally the `chrome.runtime.onMessage` listener that performs the injection) rather than continuing to guess from documentation gaps. Fixed with: `apps/extension/entrypoints/background.ts` now does a side-effect `import '@inboxsdk/core/background'` (verified present in the built bundle via `grep` for the message-type string, not just "the import didn't error"), and a new `scripts/copy-pageworld.mjs` copies the package's pre-built `pageWorld.js` into `public/` at build time — resolved via `require.resolve()`, not a hardcoded path, since npm workspaces hoists the package to the monorepo root rather than `apps/extension/node_modules`. Not committed to git (generated from node_modules on every build, stays in sync with whatever `@inboxsdk/core` version is installed). This is a real hole in the original Phase 2 InboxSDK research (ADR-3/ADR-9 verified the `Compose`/`Conversations` API surface thoroughly but never checked MV3 bootstrapping requirements) — exactly the class of gap that a clean `wxt build` and 31 passing unit tests can't catch, since neither exercises real `chrome.*` permission/injection enforcement in an actual browser.
- **ADR-13: `injectTrackingThenSend`'s fail-open catch block was completely silent — "fail open" was conflated with "fail invisibly."** After ADR-12's fix, InboxSDK loaded correctly (confirmed: the user's error logs showed our own `appId`/extension ID for the first time) but `messages` was *still* empty after a real send, with zero errors anywhere the user could see — because the catch block in `injectTrackingThenSend` was `catch {}`, by design, to satisfy NFR2 (never block a send on a tracking failure). That design goal was right; the implementation over-applied it. Measured the actual cause with `curl -w time_starttransfer`: ~620ms round-trip to the live Worker for a *trivial* auth-check-only request; `COMPOSE_INJECTION_TIMEOUT_MS` was 1500ms, and a real request (two inserts: message + link_tokens) plus the sender's own network latency to Cloudflare's edge could plausibly exceed that, causing a silent, undiagnosable `AbortController` timeout. Fixed two things together: raised the timeout to 4000ms (a few extra seconds of perceived send delay is a fair trade against tracking silently never working), and — more importantly — added `console.error`/`console.warn` calls on every skip/fail path (not signed in, tracking disabled, and the catch block itself, now with the actual error object). Fail-open still means the send is never blocked or delayed past the timeout; it no longer means a real bug is invisible to anyone, including future debugging sessions.
- **ADR-14: dwell time ("how long was it open") is explicitly out of scope — not a missing feature, a hard limitation of pixel-based tracking.** A tracking pixel is a single image fetch; it has no mechanism to report when the recipient stops looking, since email clients don't execute JavaScript that could send a "closed" signal. Every pixel-based competitor (Mailsuite, Yesware, Mailtrack.io) has the identical wall — this isn't a MailTrack gap to close, it's a category limit, and was communicated to the user as such rather than attempting a fake proxy metric (e.g., "time between fetches") that would misrepresent what's actually knowable. Open count, click count, and first/last-opened timestamps ARE real, fetch-derived signals and were built (this ADR's actual scope): `GET /v1/messages` now aggregates `verdicts` (verified_open/verified_click counts + earliest/latest open timestamp) per message via `getVerdictStatsForMessages()` — a JS-side aggregation over one page of messages at a time (bounded by `LIST_PAGE_SIZE`, 50), not a Postgres function, since unlike `classify_ip_category` this isn't on any hot path and doesn't need one.
- **ADR-15: classification moved from a once-a-minute cron sweep to inline, in the same background task as the raw-event insert.** The user reported opens/clicks taking 1-2 minutes to show up despite opening the tracked email multiple times — traced via the live `verdicts` table to the classifier sweep's cron cadence (`* * * * *`, Cloudflare's minimum granularity), which meant every fetch waited up to 60s for the next tick before it was even classified, on top of the dashboard's own (now-fixed) lack of auto-refresh. The fix: extracted the per-event classify-and-persist logic out of `sweep.ts` into `classifier/classify-and-apply.ts` (`classifyAndApplyOne`), and both `pixel.ts` and `click.ts` now call it directly, inline, immediately after logging the raw event — still entirely within the same `ctx.waitUntil()` background task that runs *after* the response is already sent, so this changes nothing about ADR-1's actual constraint (the pixel/click response itself is never slowed down). `insertRawEvent()` now takes an explicit `occurredAt` timestamp (computed once, reused for both the insert and the immediate classification call) instead of relying on the DB's `default now()`, so the classifier gets a value guaranteed identical to what's stored without a second round-trip to read it back. The cron sweep (`sweep.ts`) stays as-is, now serving as a fallback safety net for the rare case where the inline `waitUntil()` doesn't finish (Workers gives no absolute completion guarantee for `waitUntil`) — "eventually consistent via the sweep" is a much better failure mode than "silently stuck unclassified."
- **ADR-16: dashboard shows recipient, not subject, as the primary identifier.** The user pointed out a real usability gap: many tracked sends can share an identical subject line, making subject alone useless for telling them apart at a glance — recipient is the actual disambiguator. Added `messages.recipient` (plaintext, same reasoning as `subject` — the sender's own compose data shown back to the same sender, not a privacy concern) via `db/migrations/0002_add_recipient.sql` (a proper incremental `ALTER TABLE`, not an edit to `0001_init.sql`, since the database now has real live data and can't just be re-run from scratch). Extension captures it via `ComposeView.getToRecipients()` (confirmed real API, returns `Contact[]` with `name`/`emailAddress` — verified against InboxSDK's docs before use, same discipline as every other InboxSDK integration point) and a new pure `formatRecipients()` helper (joins emails, truncates beyond 3 with a "+N more" suffix). Dashboard's list view now shows recipient as the primary line with subject as a smaller secondary line beneath it, rather than dropping subject entirely — still useful context, just not the primary identifier anymore.
- **ADR-17: dashboard auto-refreshes every 5 seconds instead of requiring a manual click.** Part of the same "why isn't this updating" report — the dashboard had genuinely no polling at all (`main.ts` only ever fetched once, on page load), so even with ADR-15's faster backend classification, nothing would appear until the user manually hit "Refresh" or reloaded the page. A naive fix (tear down and rebuild the table every poll) would collapse any expanded detail row out from under whoever was reading it, which is worse UX than the original bug. Instead: `messagesById`/`rowsById` maps act as the single source of truth, and `pollRefresh()` re-fetches every currently-loaded page and updates existing rows' cells in place via a shared `paintRow()` function (used by both first render and every poll) — genuinely new messages get prepended (list is newest-first), everything else just gets its status badge/counts repainted, expanded timelines stay open. Polling pauses when `document.visibilityState !== 'visible'` (no point burning API calls against a backgrounded tab) and catches up immediately on refocus rather than waiting for the next tick. The `visibilitychange` listener is registered once at module scope, not inside `startPolling()` — an earlier draft registered it there, which would have stacked a duplicate listener on every manual "Refresh now" click.
- **ADR-18: "Read Confidence" verdict ships (Track A/E from `docs/read-detection-plan.md`); the streaming-pixel dwell-time track (Track C) is dropped, empirically, not by assumption.** The user asked for genuine read-duration detection and explicitly rejected degrading to "not solvable." Before writing any feature code, the connection-hold approach (the one real precedent, ContactMonkey's Outlook-only "Read Time Analytics") was tested directly against this stack: a temporary diagnostic route (`routes/stream-test.ts`, deployed, tested, then deleted) streamed ticks over a `ReadableStream` while `wrangler tail` watched live logs; a real request was started with curl and killed mid-stream after receiving actual bytes (proving it reached the Worker). Over 5+ minutes and past the stream's own 60-tick/120s cap, **zero trace appeared** — no `cancel()`, no thrown `enqueue()`, not even a completed-request log line — while control requests (health checks, the extension's own poll endpoint) logged instantly in the same tail session throughout, proving the tail session itself was live. This matches and extends a real, currently-open `cloudflare/workerd` GitHub issue (#5611, filed 2025-11-29): client disconnection on a streaming response is **completely unobservable** on Cloudflare Workers — not client-specific (worse than "Gmail's proxy defeats it," which was the original hypothesis), a platform-wide gap. ContactMonkey's technique depends on real TCP socket-close events, which Workers structurally doesn't expose. Presented this finding to the user with the two honest paths (drop it, or stand up a second non-Workers service later just for that one endpoint); user chose to drop Track C rather than add a new infrastructure dependency.
  Track A shipped instead, on infrastructure that already existed and needed zero new client-side mechanism: `apps/backend/src/read-signal.ts` (`computeReadSignal`, pure and unit-tested — 9 cases in `tests/read-signal.test.ts`) turns a message's ordered `verdicts` history into `ReadConfidence` (`'read' | 'likely_read' | 'glanced' | 'not_verifiable' | null`) plus a `minEngagedSeconds` **lower bound**, never an estimate: `read` = a verified click anchored to the latest preceding verified open (the gap between them is a *proven* minimum engaged time, e.g. open→click 40s apart means they demonstrably had it open for ≥40s — never reported as "open for exactly 40s"); `likely_read` = 2+ separate verified opens with no click (repeat engagement, no continuous duration claimed); `glanced` = exactly one verified open; `not_verifiable` = only `machine_suspect`/existing `not_verifiable` verdicts seen, no human open confirmed; `null` = no signal at all yet (message not opened — deliberately distinct from `not_verifiable`, which means "activity happened but wasn't human," not "nothing happened"). The originally-approved plan (see task history) called for a *new* "repeat-beacon" mechanism to catch re-opens without a click, but this turned out to be unnecessary: `pixel.ts` already sets `Cache-Control: no-store` (ADR-4, "repeat opens must be observable") and a direct production-DB query performed as part of this work confirmed real messages already carry 5+ `verified_open` rows each from Gmail re-fetching the same pixel on reopen — the "repeat-beacon" signal was already live, just not yet turned into a verdict. `getVerdictStatsForMessages()` was extended to fetch all verdict kinds (previously filtered to `verified_open`/`verified_click` only) and run `computeReadSignal` per message in the same pass that already computes `openCount`/`clickCount` — no new migration, no new endpoint. Dashboard: new "Read confidence" column (badge, mirrors the existing status-badge pattern via a new `read-confidence-chip.ts`) plus, in the expanded detail view, the full evidence sentence and (when present) an "Engaged ≥ Ns (proven minimum)" line — the evidence sentence itself is the product's honesty layer (Track E), shown, not hidden behind a tooltip alone.
- **ADR-19: Track B depth beacons ship; Track D (AMP for Email) is dropped for an architectural reason, not merely an approval-gate one.** User asked to build the two remaining tracks. Checked Track D's actual feasibility before writing any code, since "gated behind Google approval" (the plan doc's original framing) turned out to understate the real blocker: MailTrack sends by calling `composeView.setBodyHTML()` on Gmail's own web compose window and letting Gmail's own pipeline assemble and transmit the MIME message (confirmed by re-reading `inboxsdk-app.ts`'s actual send path) — there is no hook anywhere in that flow to attach the separate `text/x-amp-html` MIME part AMP for Email requires. The only way around it is switching MailTrack's entire send mechanism to the Gmail API (`users.messages.send` with a hand-built multipart message), which needs the sensitive `gmail.send` OAuth scope plus Google's AMP-sender registration — a new send architecture, not a feature addition. Presented this plainly; user's prompt went unanswered (no response to the AskUserQuestion), so proceeded on the stated recommended default: drop Track D for now, same reasoning as Track C.
  Track B shipped: `beacon_tokens` table (new migration `db/migrations/0003_add_beacon_tokens.sql`, applied to the live DB; `0001_init.sql` updated in parallel so a fresh install matches — the established two-file pattern since the DB has real data) holds two extra per-message tokens (`mid`, `bottom`), deliberately separate from `messages.pixel_token` so the original, reliable open-detection path can never be put at risk by this addition (same reasoning as ADR-15's classify-and-apply extraction). A new route (`routes/beacon.ts`, `GET /b/:token.gif`) mirrors `pixel.ts`'s exact fail-open/no-cache/no-validity-leak design and reuses the same classifier — position doesn't change what counts as a verified open, only where in the body it was seen. `raw_events.beacon_position` (`'top'|'mid'|'bottom'`) is set explicitly by both routes; `apps/backend/src/depth-signal.ts` (`computeDepthReached`, pure, 7 unit tests) resolves the deepest *verified* position reached, deliberately returning `null` for `'top'`-only history since the ordinary pixel already proves an open elsewhere (ReadConfidence) — reporting it again as a depth claim would be a second name for the same fact, not new information.
  **The critical scoping decision, made before writing the extension-side injection code:** Gmail's ~102KB "message clipped" truncation is a hard DOM fact for genuinely long messages (content past the clip point isn't in the rendered page at all until "View entire message" is clicked, so a beacon there literally cannot fetch without expansion) — but for MailTrack's overwhelmingly common case (short professional emails), a mid/bottom beacon would fire in the same batch as the ordinary pixel on every open, which would misrepresent "opened a short email" as "read to the end of a long one." That's exactly the fabrication trap ADR-14 already flagged for dwell time. Fix: `LONG_MESSAGE_BEACON_THRESHOLD_BYTES` (90,000 bytes, `routes/messages.ts`) gates beacon generation entirely — `POST /v1/messages` only returns `beaconUrls` when the extension-measured `bodyLength` clears that bar, so short sends (the default) get no beacons at all, no wasted deliverability cost, and no misleading signal. `apps/extension/src/html-injection.ts::appendDepthBeacons()` inserts the mid beacon at the next `>` boundary at-or-after the body's string midpoint (never splits inside an existing tag — 4 new unit tests, including one that deliberately places the midpoint inside an `<a href>` attribute to prove it doesn't corrupt the link) and the bottom beacon at the very end.
  **Verified against the live backend, not just unit tests**, using a disposable Supabase test user (inserted directly via SQL — `auth.users` + `public.users` — since the FK from `users.id` to `auth.users(id)` means a normal API-key-only test double isn't possible; deleted immediately after, confirmed no residue): created a message with `bodyLength` past the threshold, confirmed `beaconUrls` came back (and confirmed a short message correctly got none), fetched all three beacon positions with realistic UAs and timing (spaced past the 45s prefetch window, `GoogleImageProxy` UA for the first/top fetch per the classifier's own rules), and confirmed `GET /v1/messages` returned `depthReached: "bottom"` and a correctly-computed `readConfidence`. Confirmed the raw `beacon_position` breakdown directly in Postgres matched (one `top`, one `mid`, one `bottom`).
  **What this does NOT close out:** the *exact* Gmail clip threshold and its precise DOM-truncation behavior were reasoned from well-documented Gmail behavior, not confirmed against a live Gmail account sending/receiving through the real extension — that requires a genuinely long real email and a human to open it, the same category of test as the README's mandatory phone acceptance test (section 15), and is tracked as a follow-up rather than claimed as done.
- **ADR-20: hard-bounce detection ships; spam-folder placement and "images blocked" are refused as distinct features, not silently dropped.** User asked to detect all three of: hard bounces, spam-folder placement, and images-blocked-by-default. Assessed each honestly before building anything: spam-folder placement happens entirely on the recipient's mail server/client and produces zero notification of any kind back to the sender — there is no signal path, not a weak one, so building an indicator here would be fabricating data outright, the same category of refusal as ADR-19's AMP finding. "Images blocked" is logically indistinguishable from "not opened yet" — both produce exactly zero tracking events — so no positive state can honestly be asserted for it either; the one case where the underlying signal already exists (a click with images off, since a click needs no image fetch at all) was already covered by ADR-18's `read-signal.ts`, which classifies a click-with-no-preceding-open as `read` — nothing new needed there.
  Hard bounces ARE real: verified via the installed `@inboxsdk/core` package's own `.d.ts` source (`message-view.d.ts`, read directly rather than trusting the docs site, which turned out to be a client-rendered SPA that WebFetch couldn't extract) that `MessageView.getSender()` and `.getBodyElement()` are genuine, confirmed APIs — meaning the extension can recognize Gmail's own bounce-notification format (`mailer-daemon@...`) landing in the *sender's own inbox* and read its body text, the same watching mechanism already used for the status chip (`registerMessageViewHandlerAll`).
  Modeled as fully orthogonal to `MessageStatus`, not part of the escalate-only ladder: new `messages.bounce_detected_at`/`bounce_reason` columns (migration `db/migrations/0004_add_bounce_detection.sql`, `0001_init.sql` updated in parallel) rather than a new status value — a bounce is discovered proof the message never arrived at all, the opposite of "more confidence of engagement," so shoehorning it into the same rank-escalate mechanism would have been a category error. `apps/extension/src/bounce-detection.ts` (`isBounceSender`, `extractBounceDetails`, both pure, 9 unit tests using multiple realistic Gmail bounce body variants — "address not found," "mailbox full," and critically a temporary/delayed-delivery notice which must NOT match, since a soft bounce is often retried and later delivered) recognizes only PERMANENT failures ("failed permanently"), never temporary ones, to avoid the false-positive-turned-worse-false-negative of telling a sender their email never arrived when it may yet be retried successfully.
  Correlation (which specific tracked send does a bounce belong to) has no shared ID to key off — Gmail's bounce format doesn't echo one back — so `apps/backend/src/bounce-correlation.ts::correlateBounce()` (pure, 9 unit tests) is deliberately conservative: matches by recipient address within a 7-day plausible-delay window, and only auto-resolves when exactly one candidate remains (or is narrowed to one via a subject-excerpt match extracted from the bounce body's quoted original headers). When multiple sends to the same recipient exist in the window and can't be disambiguated, it returns `matchedMsgId: null` with an honest reason rather than guessing — a wrong bounce match would falsely tell a sender a real send never arrived, which is worse than not flagging anything.
  New `bounceDetectionEnabled` setting (default on, visible in options): reading inbox message bodies — even narrowly, only for messages whose sender already matched the mailer-daemon pattern — is a broader read than anything else this extension does (the compose hook only ever touches the sender's own draft; the status chip only ever reads message IDs), so it's surfaced as a toggle rather than silently added to existing "tracking enabled" behavior.
  **Verified end-to-end against the live backend** (not just unit tests) with a disposable Supabase test user: confirmed an unambiguous single-candidate bounce correlates and surfaces correctly on `GET /v1/messages` (including the diagnostic text appended to the reason, and `status` staying `sent` — confirming bounce is genuinely orthogonal, not overwriting the ladder); confirmed an ambiguous two-candidate case correctly withholds a match; confirmed a subject excerpt correctly disambiguates that same ambiguous case; confirmed a recipient never sent to correctly returns no match. **What this does NOT close out:** the extension-side detection itself (does `getSender()`/`getBodyElement()` behave as expected against a real bounce notification actually landing in a real Gmail inbox) has not been confirmed with a live human test — same open-follow-up category as ADR-19's clip-fold caveat, not claimed as done.
- **ADR-23: accuracy measurement harness + honest confidence display — the answer to "make it show 99% accuracy" without fabricating the number.** The user asked for a displayed accuracy figure. Named the trap directly rather than complying: you cannot *display* an accuracy percentage you have not *measured* against ground truth without fabricating the exact kind of number that makes competitors worthless — so the deliverable is a real measurement rig, not a badge. Built `apps/backend/src/accuracy/` — a hand-labeled ground-truth corpus (`scenarios.ts`, 11 real-world situations drawn from documented client behavior: Gmail proxy opens, Apple MPP prefetch by IP-range and ASN, scanner bursts, Safe-Links pre-visits, notification previews, bots — deliberately balanced across cases that MUST verify and cases that MUST NOT, since a classifier that only ever said "verified" would score 100% on a positive-only set and be useless) plus `harness.ts` (`runHarness`/`formatReport`, pure) which runs the *real* classifier over the corpus and computes a confusion matrix, per-verdict precision/recall, and an overall accuracy figure. `tests/accuracy.test.ts` runs it as a genuine regression guard: it asserts the classifier clears a 90% bar AND, separately and with zero tolerance, that no machine scenario is ever classified as a verified human open (the one error that destroys the product) — and it regenerates `docs/ACCURACY.md` on every run so the committed report can never drift from actual behavior. Current measured result: **100% (11/11)** on the labeled corpus, no misclassifications. Crucially, the report states in its own headline what that number is and is not: a measured figure for the classifier LOGIC against a labeled set, explicitly NOT a field-measured real-world rate — that requires the human-run client matrix documented in `docs/accuracy-matrix.md` (send to real Gmail/Outlook/Apple-Mail accounts, perform known actions, score verdicts; counts Apple-MPP `not_verifiable` as a correct honest abstention, not a miss), which is defined but not yet executed, same category as the phone acceptance test. Phase 3 (honest display): the dashboard now carries a "What does each confidence level actually mean?" reliability legend — Replied = definitive, Read = very high, Likely read = high, Glanced = moderate, Not verifiable = honest abstention — each with its real reason, and an explicit note that classifier accuracy is measured (see ACCURACY.md) while real-world field accuracy is a separate in-progress measurement. No flat fabricated "99%" anywhere — the product shows exactly what each signal is worth and how it was measured.
- **ADR-22: session clustering + sync-pattern detection — making "opened N times" mean what a human thinks it means.** The user, after asking whether a repeat-open could be a background mail-sync rather than a real re-read, asked for the app to be as doubt-free as possible. Two pure functions in `apps/backend/src/open-analysis.ts` (10 unit tests): `clusterOpenSessions` collapses verified-open timestamps within a 30-minute gap into one *viewing session* (rapid re-fetches from a single sitting stop being counted as separate "opens"), and `detectSyncPattern` flags a suspiciously clockwork-regular series (3+ opens, coefficient-of-variation < 0.15, mean interval wider than the session gap) as consistent with an automated poll rather than a human returning. The two are complementary: a mail app polling every few minutes collapses into one session automatically (so it can't inflate the count), while one polling on a wider fixed schedule (e.g. every 45 min) would register as many sessions — which is exactly what the sync detector catches. Crucially, `computeReadSignal`'s confidence branch now keys on *session count*, not raw open count: five fetches in one sitting is `glanced` (one viewing occasion), not `likely_read` — the direct fix for the "opened 5 times?" overstatement the user flagged. `syncSuspect` is surfaced as an honest caveat appended to the evidence sentence, never used to silently downgrade a verdict. New `sessionCount`/`syncSuspect` fields flow through `MessageSummary` to the dashboard ("Opened 5 times across 3 sessions"). **Verified end-to-end against the live backend** by inserting verdict rows at the user's actual real-world timeline (opens at +29/+44/+56/+118/+535 min) via direct SQL and confirming the deployed `GET /v1/messages` returned `openCount: 5, sessionCount: 3, readConfidence: likely_read` with the "3 distinct sessions" evidence — the exact scenario from the conversation, now reported honestly.
- **ADR-21: reply detection — the one signal that removes the read/glance/sync ambiguity entirely.** Prompted by the user pushing on a real weakness: a pixel "open" can't distinguish a human reading from a background mail-sync re-rendering the message, and there's no honest fix for that at the pixel level (same wall as ADR-14's dwell time). A *reply*, however, cannot be produced by any sync, proxy, or prefetch — it requires a human to author it — so it's the strongest possible read-proof, and it was the highest-value of the accuracy improvements the user asked for. Verified `MessageView.getThreadView().getThreadIDAsync()` and `getSender()` against the installed package's own `.d.ts` (`thread-view.d.ts`, `message-view.d.ts`) before building, same discipline as ADR-20. Modeled as the new top of the escalate-only status ladder (`sent < delivered < opened < clicked < replied`, `classifier/escalation.ts`) rather than an orthogonal flag like bounce — because unlike a bounce (non-delivery, the opposite of engagement), a reply IS the strongest form of engagement escalation and fits the ladder's meaning exactly. New `messages.reply_detected_at` column + a widened status CHECK constraint (migration `0005`, `0001_init.sql` updated in parallel). Correlation is trivial here, unlike bounce: the extension stores a `threadId -> {msgId, recipientEmails}` map at send time (captured from `getToRecipients()` at presending, recorded on the `sent` event via `getThreadID()`), so when a message later renders in that thread whose sender is one of the original recipients, it's unambiguously a reply to a known msgId — `POST /v1/replies` just validates ownership and escalates, no fuzzy matching. The sender-in-recipients check cleanly ignores the sender's own follow-ups (their address is never in the recipient list) without needing to know the account's own address. Reply detection reads only sender + thread ID, never body text (unlike bounce), so it needs no separate privacy toggle. A reply also overrides read-confidence to `read` with the definitive evidence sentence "Replied to your email — definitive proof they read it," since it outranks any pixel/click inference. **Verified end-to-end against the live backend** with two disposable test users: confirmed reporting a reply escalates status to `replied`, surfaces the `reply` info, and overrides `readConfidence` to `read`; confirmed a second user gets an opaque 404 trying to mark the first user's message replied (ownership enforced); confirmed body validation rejects a malformed request. **What this does NOT close out:** the extension-side thread-watching itself against a real reply landing in a real Gmail thread is a human-required test, recorded as an open follow-up (same category as ADR-19/ADR-20's caveats).

## 6. API Design

Base URL: `https://api.mailtrack.dev` (placeholder domain until purchased)

| Method | Path | Purpose | Auth |
|---|---|---|---|
| POST | `/v1/auth/provision` | Exchange a Supabase session for a MailTrack API key (issues/rotates); see ADR-10 | Supabase access token |
| GET | `/auth/confirmed` | Static landing page for Supabase's email-confirmation redirect; see ADR-11 | none (public) |
| POST | `/v1/messages` | Register a new tracked send; returns `{msgId, pixelToken, linkToken}` | API key (per-install) |
| GET | `/v1/messages?offset=<n>` | Paginated list of the user's tracked messages (subject, status, sentAt) — backs the dashboard message list (M5) | API key, owner-scoped |
| GET | `/p/:token.gif` | Tracking pixel. Always returns a 1x1 gif regardless of token validity (fail open, no errors leak state) | none (public) |
| GET | `/l/:token` | Click redirect. 302 to original URL, logs click event | none (public) |
| GET | `/v1/messages/:msgId/events` | Timeline of raw + classified events for a message | API key, owner-scoped |
| GET | `/v1/messages/:msgId/status` | Current verdict only (`sent\|delivered\|opened\|clicked\|not_verifiable`) | API key, owner-scoped |
| GET | `/v1/events/poll?since=<iso>` | Verdict upgrades (opened/clicked) since a timestamp, across the user's messages — see ADR-7 (replaces the originally-planned SSE stream) | API key, owner-scoped |
| DELETE | `/v1/messages/:msgId` | Delete tracking data for a message | API key, owner-scoped |
| GET | `/v1/messages/:msgId/export` | CSV export of event timeline | API key, owner-scoped |

All endpoints return JSON except `/p/*` (image/gif) and `/l/*` (302 redirect) and `/export` (text/csv).

## 7. Database Schema

Canonical, executable source: [`db/migrations/0001_init.sql`](./db/migrations/0001_init.sql) — this section is a summary, not a duplicate; if they ever disagree, the migration file wins.

- `users` — one row per signed-up account (v1 has no multi-seat orgs). `id references auth.users(id)` — the Supabase Auth user's own id, not a separate generated UUID (ADR-10). `api_key_hash` only, never plaintext.
- `messages` — one row per tracked send. `status` is the escalate-only ladder value (`sent|delivered|opened|clicked|not_verifiable`), `pixel_token` is the unique 128-bit token embedded in the pixel URL, `subject` is plaintext (the sender's own subject line, shown back to the same sender in the dashboard — not a privacy concern, so not hashed; an earlier `subject_hash` column was dead code, never written or read).
- `link_tokens` — one row per rewritten link in a message, maps `token → original_url`.
- `raw_events` — every fetch/click, unfiltered, append-only. Carries `ip_hash` (never raw IP, NFR4), `asn` (from `request.cf.asn`, ADR-6), `ip_category` (resolved inline at log time via `classify_ip_category()`, ADR-8), and `classified_at` (null until the cron sweep processes it — lets the sweep cheaply select the pending queue).
- `verdicts` — classified, append-only; this is what drives status changes and notifications. Carries a human-readable `reason` shown directly in the UI timeline.
- `asn_intel` — `asn → category` mapping for security-scanner ASNs (Proofpoint/Mimecast/Barracuda/etc). Populated from verified vendor documentation, never guessed (see Known Issues).
- `ip_ranges` — `cidr → category` mapping, currently Apple Private Relay egress ranges refreshed weekly from Apple's published CSV. Queried via the native `inet <<= cidr` containment operator wrapped in `classify_ip_category(inet) returns text`, longest-prefix-match wins (ADR-8).
- RLS is enabled with no policies on every table as defense-in-depth — these tables are only ever touched with the service-role key, never the anon key.

## 8. Chrome Extension Architecture

- **Framework:** WXT (Vite-based MV3 tooling) + TypeScript. Builds cleanly (`wxt build`) into a valid MV3 bundle — verified during Phase 2 implementation.
- **Gmail integration:** `@inboxsdk/core`, loaded from the `gmail.content.ts` content script (matches `https://mail.google.com/*`).
- **Compose hook:** `sdk.Compose.registerComposeViewHandler` — see ADR-9 for the exact send-interception pattern (cancel-then-resend, not "modify and let it proceed"). On the first `presending`, calls `POST /v1/messages`, rewrites `<a href>` tags in the compose HTML to their tracked redirect URLs, appends the invisible pixel `<img>`, writes it back via `composeView.setBodyHTML()`, then calls `composeView.send()` itself.
- **Thread view hook:** `sdk.Conversations.registerMessageViewHandlerAll` — for each rendered message, resolves its Gmail message ID (`messageView.getMessageIDAsync()`), looks it up against the local Gmail-ID→msgId map (populated from the compose hook's `sent` event, since the real Gmail ID isn't known until then), fetches current status, and renders `messageView.addAttachmentIcon({ iconUrl, tooltip })` — confirmed real API, not a guessed one.
- **Background service worker:** `chrome.alarms` (1-minute floor, MV3 clamps `periodInMinutes`) calls `GET /v1/events/poll` (ADR-7 — polling, not SSE) and fires `chrome.notifications.create` only for `opened`/`clicked` upgrades in the response, which by construction (ADR-5) are always verified.
- **Options page:** email+password signup/login (calls Supabase Auth directly with the public anon key, then exchanges the session for an API key via `POST /v1/auth/provision` — ADR-10) as the primary flow, with a collapsed "Advanced: enter an API key manually" fallback for troubleshooting. Once signed in: default-tracking toggle, notification toggle, sign-out, a link to the dashboard, CSV export by message ID, delete-tracking-data by message ID (FR9/FR10).
- **Dashboard page (M5):** a separate `dashboard.html` extension page (not embedded in options — a full list view needs the room). Paginated message list (`GET /v1/messages`, newest-first) with a status dot + subject + sent time per row; clicking a row expands an inline timeline (`GET /v1/messages/:msgId/events`) with suppressed/machine-classified events visibly greyed out rather than hidden (FR5), each showing its human-readable classifier reason. That reason string can contain attacker-influenceable content (e.g. a raw `User-Agent` header echoed into a `machine_suspect` reason) since it's rendered via `innerHTML` for inline formatting — escaped through a `textContent`-round-trip helper before interpolation, the one real XSS-relevant surface found on this page.
- **Fail-open guarantee (NFR2):** `injectTrackingThenSend()` wraps the entire tracking attempt in try/catch/finally; `composeView.send()` is called in the `finally` block unconditionally, so a network failure, timeout, 4xx/5xx, or missing API key all fall through to an untracked send rather than blocking or losing the email.

## 9. Backend Architecture

- **Runtime:** Cloudflare Workers, `Hono` router.
- **Pixel handler (`/p/:token.gif`):** returns a static 43-byte GIF synchronously, identical regardless of token validity (no enumeration signal). All DB work — token lookup, IP-range/ASN resolution, `raw_events` insert — runs in `ctx.waitUntil()` after the response is already flushed, so it can never add latency to the image fetch (ADR-1, NFR1).
- **Click handler (`/l/:token`):** must resolve the token before it can redirect (no safe fallback), so that one lookup is on the hot path; the `raw_events` insert itself still runs in `waitUntil()`.
- **Classifier sweep:** scheduled Worker cron (`* * * * *`, every minute) reads `raw_events where classified_at is null`, resolves prior-fetch context (first-fetch / burst count) and ASN/IP-range intel, calls the pure `classifyEvent()` function, writes a `verdicts` row, and advances `messages.status` through the escalate-only ladder. Queues would allow lower latency but require a paid Cloudflare plan; documented as a revisit if the 1-minute cron's ~60s worst-case classification lag ever fails NFR7 in practice.
- **Intel refresh:** weekly scheduled Worker (`0 3 * * 1`) re-downloads Apple's published egress-range CSV into `ip_ranges`. Security-scanner ASN data (`asn_intel`) is written by a separate, manually-invoked upsert path — no data ships until it's sourced from verified vendor docs (see Known Issues).
- **Data layer:** `@supabase/supabase-js`, which is fetch-based and works natively in Workers — no Hyperdrive or raw TCP connection needed (resolves the open decision from the original plan).

## 13. Security Checklist

- [x] All tokens (pixel, link, API key) are cryptographically random, ≥128 bits (`lib/crypto.ts::randomToken`, 16 bytes via `crypto.getRandomValues`).
- [x] API keys are stored hashed (never plaintext) — matches `users.api_key_hash`, SHA-256 via Web Crypto.
- [x] IP addresses hashed at rest; raw IP never written to `raw_events` unhashed (`ip_hash` column, raw IP only touched transiently in the pixel/click handler to derive `ip_category` and the hash).
- [x] Pixel/click/message-creation endpoints rate-limited via Cloudflare's native Rate Limiting binding (declarative in `wrangler.toml`, no external KV namespace to provision — verified real syntax before writing it). Pixel/click are keyed by requester IP and fail soft (skip logging, never change the response the requester sees — consistent with the fail-open/no-validity-leak design elsewhere); message creation is keyed by user and returns a normal 429, bounding a leaked API key's blast radius to 30 sends/minute.
- [x] CORS locked to `ALLOWED_EXTENSION_ORIGIN` for all `/v1/*` routes via `hono/cors`, applied per-request (env bindings aren't available at module scope in Workers). Falls back to permissive `*` until the real `chrome-extension://<id>` is known post-publish (Known Issues) — a loud, documented placeholder rather than either a hardcoded guess or a silent no-op.
- [x] No user-supplied data reflected unescaped (XSS) in any dashboard view. Options page (M3) uses `textContent`/`.value` only, never `innerHTML` with server data. Dashboard page (M5) does use `innerHTML` for inline event formatting — its one field that can carry attacker-influenceable content (`event.reason`, which can embed a raw `User-Agent` header) is passed through an `escapeHtml()` helper (round-trips via `textContent`) before interpolation; the other interpolated fields (`kindLabel`, `event.verdict`) come from closed backend enums, not free text, so they don't need escaping.
- [x] Redirect endpoint only ever redirects to `original_url` looked up server-side by token; there is no client-supplied URL parameter, so open-redirect via tampering is structurally not possible.
- [x] Extension API key stored in `chrome.storage.local` (extension-private storage, not accessible to Gmail's own page scripts) rather than `localStorage`, which the content script's page-world execution could expose to Gmail's page context.
- [ ] Dependency audit clean before release. `npm audit` reports 14 vulnerabilities (5 moderate/5 high/4 critical, down from 18 after the wrangler 3-to-4 upgrade above) across `wrangler` and `wxt`'s transitive dev/build dependencies — not in shipped runtime or bundled extension code (confirmed via `wxt build` output inspection: bundled content script only pulls in `@inboxsdk/core` and our own source). Deferred to before M7 release; re-audit then since both tools ship frequent patches.
- [x] Secrets (Supabase keys, MaxMind license) referenced only via `Env` bindings / `wrangler secret put`, never present in any committed file — verified via `.gitignore` (`.env*`) and `wrangler.toml` containing only non-secret `[vars]`.
- [x] `POST /v1/auth/provision` (ADR-10) never trusts a client-supplied identity — the Supabase access token is validated server-side via `auth.getUser(jwt)` against Supabase's own Auth service before any `users` row is touched; there is no path where a caller can claim to be a given user without a token Supabase itself issued. Rate-limited separately (`AUTH_RATE_LIMITER`, keyed by IP) from message creation, so account-creation/key-rotation abuse can't be laundered through the messages quota or vice versa.
- [x] The Supabase anon key embedded in the extension (`config.ts::SUPABASE_ANON_KEY`) is Supabase's own public-by-design client key — safe to ship in extension source, access is enforced by Supabase's RLS policies and by `/v1/auth/provision`'s server-side token validation, not by keeping this value secret.

## 14. Testing Strategy

- **Unit tests:** classifier rules in isolation (pure functions, fixture-driven) — fastest feedback loop, run on every commit.
- **Integration tests:** Hono routes against a test Supabase instance (or local Postgres via Docker) — verifies raw_events are written correctly and pixel/click handlers fail open.
- **E2E tests:** Playwright driving a real Gmail compose + send in a test account, verifying pixel/link injection actually lands in the sent message.
- **Regression tests:** see below — permanent, never removed.

## 15. Regression Tests (permanent)

Two tiers: a manual live-device test that is the ultimate source of truth, and an automated unit suite (`apps/backend/tests/classifier.test.ts`, `describe('regression: permanent fixtures')`) that exercises the same six scenarios against the pure classifier functions on every commit. The unit suite passes today (22/22); the live-device test can't run until the backend is deployed (see Known Issues) but its steps are specified now so it's ready the moment deployment credentials exist.

1. **The mandatory phone acceptance test (manual, live device):** send a tracked email to a real device inbox, confirm notification-preview-only fetch does NOT upgrade status past `delivered`, confirm actual open upgrades to `opened` (verified), confirm link click upgrades to `clicked`. Run per release until a device-farm harness exists. Unit-level proxy: `classifier.test.ts` fixture 1.
2. Apple MPP fixture: pixel fetch from a known Apple Private Relay egress range within 5s of send never escalates to `opened` (now IP-range based per ADR-8, plus an ASN-based fallback fixture).
3. Scanner-burst fixture: 10 resource fetches within 500ms of delivery never escalates past `delivered`.
4. Repeat-fetch fixture: pixel fetched once at delivery (machine pattern) then again 2 hours later (human pattern) escalates to `opened`.
5. Click fixture: a link click from an Apple Private Relay egress range still escalates to `clicked` (Private Relay doesn't auto-follow links, unlike its pixel-prefetch behavior — ADR-8/rules.ts). A link click pre-visited by a known security-scanner ASN or user-agent does NOT escalate (Microsoft Safe Links / Proofpoint URL Defense / Mimecast auto-scan rewritten links before the recipient opens the email — the click-side analog of the pixel prefetch problem, caught in the M6 security review and fixed before it shipped as a permanent false-positive).
6. Verdict-never-downgrades fixture: once `opened`, a later `machine_suspect` classification does not revert status.

- Built the correlation logic (`bounce-correlation.ts`) deliberately conservative rather than maximally clever: no shared ID exists between a bounce notification and the original send, so it matches by recipient address within a time window and only auto-resolves when the result is unambiguous (or narrowed to one by a subject-excerpt match) — an ambiguous multi-candidate case returns "no match" with the reason stated, rather than picking the statistically-likely one and risking telling a sender a real, successful send never arrived. Wrote the regex extraction (`bounce-detection.ts`) against multiple realistic Gmail bounce-body variants, not one crafted example — including a deliberate negative case (a temporary/delayed-delivery notice) to confirm soft bounces are never misreported as hard ones, since a soft bounce is often retried and later delivered successfully.
- Added a new, visible `bounceDetectionEnabled` setting (default on) rather than folding this into existing tracking-enabled behavior, since reading inbox message bodies — even narrowly scoped to mailer-daemon-sender messages only — is a broader read than anything else the extension does today.
- **Verified end-to-end against the live backend**, not just unit tests: created a disposable Supabase test user (same direct-SQL pattern as ADR-19, cleaned up after), then exercised all four real scenarios through the actual deployed `/v1/bounces` endpoint — unambiguous single match, ambiguous multi-candidate withheld correctly, subject-excerpt disambiguation, and a recipient with no tracked send at all — confirming each returned the correct `matchedMsgId` and that a successful match left `status` untouched at `sent` (proving bounce really is orthogonal to the ladder, not overwriting it) while still surfacing correctly on `GET /v1/messages` with the diagnostic text included.
- Full verification: 52 backend tests (43 + 9 new `bounce-correlation.test.ts`) and 52 extension tests (42 + 9 new `bounce-detection.test.ts` + 1 new dedup case in `storage.test.ts`) passing, both `tsc --noEmit` clean, extension `wxt build` clean, backend deployed live. As with ADR-19's clip-fold caveat, explicitly did not claim the extension-side inbox detection itself as confirmed against a real bounce landing in a real Gmail inbox — that's a human-required test, recorded as an open follow-up.
- **User pressed on a genuine weakness — "did they really read it, or did Gmail just re-render and we counted it?" — then asked to make the app 99% accurate.** Answered the honesty trap head-on first: we cannot *display* "99% accurate" without *measuring* accuracy against ground truth, or we'd be fabricating the exact number that makes competitors worthless. Reframed "99%" into a real three-phase plan: (1) add signals that remove the remaining doubt, (2) build a measurement rig that produces a defensible accuracy number, (3) display per-signal reliability honestly. User chose full scope.
- **Built reply detection (ADR-21) first — the single highest-value accuracy lever.** A reply is the one signal immune to the sync/proxy ambiguity the user identified, because it requires a human to write it. Verified the InboxSDK threading API against the installed package's `.d.ts` before building (thread-view/message-view), modeled `replied` as the new top of the escalate-only ladder (it fits the ladder's meaning, unlike bounce which is orthogonal), and kept correlation trivial by storing a threadId->msgId+recipients map at send time rather than fuzzy-matching. Verified end-to-end against the live backend across three scenarios (escalation + read-confidence override, cross-user ownership 404, body validation) with disposable test users, cleaned up after.
- **Built session clustering + sync detection (ADR-22) second — directly resolving the "opened 5 times?" ambiguity the user raised.** Rather than treat every pixel fetch as a separate "open," collapsed fetches within 30 minutes into one viewing session and switched the confidence logic to key on session count, so rapid re-renders in one sitting read as `glanced` (one occasion) instead of `likely_read`. Added a sync-pattern detector for the harder case (clockwork-regular polling that spans session gaps) that surfaces as an honest caveat, never a silent reclassification. Verified end-to-end by replaying the user's own real timeline (5 opens → 3 sessions) through the deployed backend.
- **Built the accuracy harness (ADR-23) third — and refused the literal ask ("display 99%") in favor of the honest one.** Called out that a displayed accuracy number you haven't measured is fabrication, then built the thing that makes a number real: a labeled ground-truth corpus + a harness that runs the actual classifier over it and produces a measured confusion matrix (100% / 11 scenarios), guarded in CI with a zero-tolerance test against the one fatal error (a machine scored as a human open), regenerating `docs/ACCURACY.md` on every run so it can't drift. Was careful to state, in the report and the dashboard both, exactly what the number is (classifier logic vs. a labeled set) and is not (a field-measured real-world rate) — the latter needs the human-run client matrix I defined in `docs/accuracy-matrix.md`. Phase 3 shipped an honest per-signal reliability legend in the dashboard instead of a flat "99%" badge. This closes the user's request the only way that doesn't undermine the product's entire premise.
